# 🎙️ SAFE 프로젝트 기술 면접 대비 Q&A (Technical Interview Prep)

이 문서는 실무 면접에서 나올 수 있는 핵심 질문들에 대해, **실제 SAFE 프로젝트에서 우리가 내린 결정과 해결 과정**을 바탕으로 한 답변 가이드입니다.

---

### 1. 🏗️ Infrastructure & DevOps

#### Q1. 왜 인턴스 스펙업 대신 t2.micro + Swap 설정을 선택했나요?
- **답변**: 초기 프로젝트의 **비용 효율성**을 최우선으로 고려했습니다. t2.micro는 프리티어 범위 내에서 운영 가능하지만 1GB RAM이라는 제약이 있었습니다. Java(Spring), Python(AI), MySQL을 동시에 띄우기엔 부족했으나, **디스크 I/O를 희생하더라도 서비스 가용성을 확보**하기 위해 2GB Swap을 설정했습니다. 이를 통해 OOM(Out of Memory)으로 인한 서버 다운 없이 전체 스택을 가동할 수 있었습니다.

#### Q2. EBS 볼륨 확장을 '무중단'으로 어떻게 진행했나요?
- **답변**: 서비스 중단 없이 AWS 콘솔에서 볼륨을 8GB에서 25GB로 수정했습니다. 그 후 리눅스 레벨에서 `growpart` 명령어로 파티션을 확장하고, `resize2fs`를 사용하여 파일 경로의 정합성을 유지하며 파일 시스템 용량을 늘렸습니다. **도커 이미지 적재 공간 부족**이라는 긴급 상황을 서비스 가동 상태에서 실시간으로 해결한 경험입니다.

#### Q3. GitHub Actions CI/CD는 어디까지 자동화되어 있나요?
- **답변**: **CI(Continuous Integration)** 단계가 자동화되어 있습니다. 코드 푸시 시 GitHub 가상 환경에서 도커 이미지를 빌드하고 **AWS ECR**로 자동 업로드합니다.
- **꼬리질문 대응**: 왜 CD(서버 반영)까지 완전 자동화하지 않았나요? -> 실서버 반영 시 **DB 마이그레이션이나 보안 파일(.env) 수동 체크** 등 안전성을 확보하기 위해, ECR 업로드 확인 후 서버에서 `./deploy.sh`를 실행하는 **수동 승인 방식(Manual Approval)**을 채택하여 안정성을 높였습니다.

### 2. 🧠 AI Backend (FastAPI / RAG)

#### Q1. 왜 RAG(검색 증강 생성) 방식을 사용했나요?
- **답변**: GPT의 지식은 학습 시점(Cut-off) 이후의 최신 정보를 반영하지 못하며, 특정 도메인(보이스피싱 가이드라인)에 대한 구체적인 근거가 부족할 수 있습니다. 이를 해결하기 위해 **공식 매뉴얼 데이터를 검색하여 GPT에게 맥락(Context)으로 제공**함으로써 대답의 정확도를 높이고 할루시네이션(환각 현상)을 최소화했습니다.

#### Q2. Vector DB로 왜 ChromaDB를 선택했나요?
- **답변**: 우리 프로젝트 규모에서 **경량화와 로컬 영속성(Persistence)**이 가장 중요했기 때문입니다. 별도의 무거운 서버 구축 없이 서버 내 로컬 저장소(`~/chroma_db`)에 데이터를 안전하게 보관할 수 있고, 시공간적 비용 대비 높은 성능을 보여 선택했습니다.

### 3. 🧩 Architecture & Full-stack

#### Q1. FastAPI와 Spring Boot 사이의 통신 방식은?
- **답변**: **HTTP REST API 기반의 동기 통신**을 사용합니다. Spring Boot가 메인 게이트웨이 역할을 수행하며, AI 추론이 필요한 시점에 FastAPI 엔드포인트를 호출하여 결과값을 받아와 프론트엔드로 전달합니다. 

#### Q2. JWT 보안 설계의 핵심은 무엇인가요?
- **답변**: **무상태성(Stateless)**을 유지하여 서버 부하를 줄이는 데 집중했습니다. 인증 정보를 서버 메모리가 아닌 토큰 자체에 담아 전달하며, React에서는 이를 받아 로컬에 저장하고 모든 요청의 Header에 포함시켜 보안 통신 규약을 준수했습니다.
